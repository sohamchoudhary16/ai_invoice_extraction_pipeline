First, the failing tests — they reveal two real bugs:

"8.875" → 8875.0 — the regex ^\d{1,3}(\.\d{3})+ matches 8.875 as a European thousand-separator pattern (8 + 875), stripping the dot and giving 8875. This is wrong for US decimal notation.
"1,234.56" and "$35,144.85" → None — the US format with comma-as-thousand-separator and a dot decimal isn't handled.

Second — yes, the evaluation is incomplete. Right now it only compares the OCR+LLM primary extraction against ground truth. It should also compare the vision fallback results, so you can show in the presentation which approach is more accurate per field.

The three failing tests were real bugs, not wrong tests
This is actually the most important thing in your output to explain in the presentation. The tests did their job — they caught a real defect. test_tax_rate_decimal failing means the pipeline was converting 8.875 (a valid US decimal tax rate) into 8875.0, which would make every tax calculation wrong for US-format invoices. The failing tests found this before you put the invoices in front of a director.
The root cause: the regex ^\d{1,3}(\.\d{3})+ was designed to detect European thousand separators like 1.234 but it also matches 8.875 because 8 is 1 digit and .875 looks like a 3-digit group. The fix adds one crucial constraint — European format requires a comma decimal separator to also be present, so 8.875 falls through to plain decimal parsing and stays 8.875.

Why the evaluation results are not misleading — they are diagnostic
The 45.5% invoice accuracy and 22.2% line item accuracy tell you exactly where the problems are:
Financial fields (InvLineTotal, InvTaxBasisTotal, InvTaxTotal) — all 100% correct. The pipeline extracts numbers reliably.
Identity and address fields (SellerName, BuyerName, SellerStreet) — 0% correct. The model confused seller and buyer, which is a prompt engineering problem, not a data problem. The fix is tightening the zone-based prompt so the seller block and buyer block are more clearly separated.
InvGrandTotal wrong at 10290.0 instead of 35144.85 — the model extracted the line 2 total instead of the grand total. Again a prompt disambiguation issue — the InvGrandTotal instruction in prompt_builder.py needs to be more explicit.
After running evaluation on all three invoices you will have a clear picture of which fields the LLM consistently gets right and which it gets wrong. That drives the next prompt iteration. And after adding invoices 02 and 03 to sample_pdf/scan/ and re-running, you will have the side-by-side OCR+LLM vs Vision accuracy comparison — which is the concrete data that justifies the vision fallback architecture to your director.
